Background:

This is a project I have been wanting to make for myself for a long time. As a UX engineer in previous roles, logging accessibility issues is a pain. I wanted to make a tool that would help me log accessibility issues and then have a system that would automatically generate a report for me.

Problem:

Logging accessibility issues is usually a manual process. There are many tools that can help, but they are not always easy to use. Spreadsheets, the most common tool, but they are tedious, especially when you have a lot of issues.

Furthermore, non-technical people don't often know what to make of the long lists and details, so further explanation is required.

Here is a discussion on Reddit that talks more about these issues: Logging accessibility issues

Agentic System Design:

At my current role, I have been using a makeshift system of AI to log issues and generate reports. I was doing this manually just using a few different instances in ChatGPT. It occured to me that others might want to use this as well so I began to workout the experience of how that would be in a unified UI.

Based on how I was using AIs already, I decided to go with a agentic system similar to the Parallelization workflow coined by Anthropic.

I chose a candidate with strong leadership skills and a solid background in user research, essential for supporting the company’s unique and disruptive approach to the jewelry industry.

We planned to bring on three more designers after the discovery phase. For now, our focus was on developing a research strategy tailored to this specific use case.

In my case, there will be 4 Agents:

  • LLM 1 - for taking users issue descriptions and providing more detail, suggesting fixes, and assigning WCAG standards.
  • LLM 2 - will be used to generate reports. This will include a summary of the issues, a list of fixes, and a detailed report.
  • LLM 3 - will summarize the impact on disabled users. This would be extremely helpful to designers to help them learn user experience as it applies to people with disabilities.
  • LLM 4 - would be the final LLM (The Aggregator) that would create VPATs from all of this information. VPATs are extrememlly expensive (can be $10000+) and time consuming, often taking weeks to months to complete. This would cut down the time and money spent on VPATs considerably.


Object Design:

The object structure of this application would all users to categorize issues at a few levels.

  • Issues - This is the base element that everything is built on top of. Users create issues logging individual accessibiity problems.
  • Assessments - Assessments are used to group issues together. For example, if you have a large website, you may want to break it down in parts like signup or transactions. Reports can be generated from this level.
  • Projects - Projects are used to group assessments. You might have many assessments for one Project. Reports can be generated at this level as well.


UI Design:

The UI for this application has to be very, very simple. Think JIRA, but even more stripped down. The idea being that anyone (Technical and non-technical) should be able to use this to log issues they find.



Wireframe #1: Issue List




When creating the hifidelty designs, it was important to make sure that the UI was itself, accessible. Large bold fonts, very clear focus/hover states, and high contrast colors were all used to ensure that the UI was accessible to users with visual disablities.

Special attention was also paid to keyboard interactivity. As mentioned before, logging issues can be tedious so every extra comfort that could be given to the users was consider in this design.



Technical:

The stack I went with was NextJS, TailwindCSS for the frontend and FastAPI and PostgreSQL for the backend. The NextJS server would uses GraphQL to communicate with the backend.

The Issue Enrichment LLM would be done using NextJS as there is not much computation needed and there's no need to send information through the whole pipeline. Also, issue images would be stored in Cloudinary with the URLs for those images being stored in the database. This would all be handled by NextJS.

The remaining Agents would be used via a FastAPI server since these will be happening asychronosly behind the scenes and require much more computation. When the user views the reports/summaries for assessments, the infomration would have already been enriched and ready for them to read.



Conclusion:

The first part of this application is built. Users can log in and log issues using the Issue Enrichent AI. I plan to put up a live demo with credentials soon. User testing is next before I continue because ensuring that logging fast, efficient, and accurate is the core of this application.
